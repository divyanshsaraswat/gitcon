# v1: Agent Context in Git — Design Proposal

## The Problem

Git solves *what* changed and *when*. Commit messages partially solve *why*. But in agentic development — where an AI makes dozens of micro-decisions per session (why this algorithm, why this interface shape, why this tradeoff) — those decisions evaporate. The diff survives. The reasoning doesn't.

Code review suffers because reviewers see the output of thinking, not the thinking. Future agents re-derive decisions that were already made, often differently. The larger the codebase, the worse this gets.

The goal: carry decision context as a first-class citizen through Git's own object model, so context travels with the code, is content-addressed like the code, and is as reviewable as the code.

---

## The Key Insight: Git Notes

Git already solved the "attach metadata to a commit without mutating its SHA" problem. It's called **`git notes`**, implemented in [builtin/notes.c](../builtin/notes.c).

Notes work like this:
- A commit `abc123` gets a note by storing a blob in a parallel ref namespace: `refs/notes/<namespace>`
- The note ref points to a tree where entries are named by commit SHA
- The commit SHA never changes — notes are a sidecar, not embedded
- Notes push/fetch like any other ref

The proposal is to use `refs/notes/context` as the canonical namespace for agent/human decision context, structured as JSON blobs stored in Git's own object database.

Context becomes just another object in the DAG — content-addressed, integrity-verified, packable, transportable — using zero new Git primitives.

---

## How Git's Object Model Maps to This

```
Git Object DB (object-file.c, odb.c)
│
├── blob: "abc123..."   ← your source file
├── blob: "{...json}"   ← a context object  ← NEW (same storage)
│
├── tree: [...files...] ← your working tree
├── tree: [...notes...] ← note tree (commit-sha → context blob)
│
├── commit: abc123      ← your commit (SHA unchanged)
└── ref: refs/notes/context  ← points to the note tree
```

Nothing new at the storage layer. The object database (`odb.c`, `loose.c`, `packfile.c`) handles context blobs the same way it handles source files. Deduplication, compression, pack transport — all inherited for free.

## The Context Object Schema (v1)

Each context is a JSON blob stored as a Git object. Minimal schema:

```json
{
  "v": 1,
  "kind": "decision",
  "author": "agent",
  "agent": {
    "model": "claude-sonnet-4-6",
    "session": "sess_abc123"
  },
  "summary": "One-line: what was decided",
  "rationale": "The full reasoning. Why this approach over alternatives. What constraints drove it. What was uncertain.",
  "decision_class": "architecture | algorithm | interface | tradeoff | bug-fix | experiment",
  "mode": "deciding | exploring | implementing | debugging",
  "alternatives": [
    { "option": "what else was considered", "rejected_because": "why not" }
  ],
  "constraints": ["must be O(1)", "must not break wire protocol"],
  "confidence": 0.85,
  "uncertainty": "what remains unresolved or assumed — the honest gap in this decision",
  "affects": ["path/to/file.c", "path/to/other.h"],
  "related": ["<sha-of-another-context-blob>"]
}
```

Two fields deserve attention:

- **`related`**: the context graph edge — links this decision to a prior one by the SHA of the other context blob. Since blobs are content-addressed, this is a stable, verifiable reference across time and machines. It can't silently drift.
- **`uncertainty`**: what the agent explicitly did *not* resolve. This field is load-bearing for v2 — unresolved uncertainties thread across sessions and surface as warnings before the area is touched again.

## Storage Layout

```
.git/
├── objects/            ← context JSON blobs stored here (via object-file.c)
│   └── ...
└── refs/
    └── notes/
        └── context     ← ref pointing to the note tree
```

The note tree under `refs/notes/context` is a standard Git tree. Each entry:
- **Name**: full 40-char hex SHA of the commit it annotates
- **Object**: SHA of the JSON context blob

This is exactly what `git notes` already produces. No new storage format needed.

## The Tooling Layer: `git ctx`

A new porcelain command (or shell script in `contrib/`) that wraps `git notes` with context-specific UX:

### Writing context

```bash
# After a commit, attach context to HEAD
git ctx add --rationale "Chose reftable over packed-refs here because
packed-refs has O(n) read for large ref counts. reftable gives O(log n)
via block index. Trade-off: reftable is not human-readable on disk."

# Or pipe structured JSON directly
echo '{"summary":"...", "rationale":"..."}' | git ctx add --stdin
```

Internally: `git notes --ref=context add -F <tempfile> HEAD`

### Reading context

```bash
git ctx show HEAD              # show context for one commit
git ctx show abc123            # by SHA
git ctx log                    # git log but with context inline
git ctx log --oneline          # summary: SHA + one-line rationale
git ctx trace HEAD             # walk the related graph backwards from a commit
git ctx trace path/to/file.c   # decision history for a specific file
```

### Syncing with remote

```bash
git ctx push origin            # push context to remote
git ctx fetch origin           # pull context from remote
```

This is just:
```bash
git push origin refs/notes/context:refs/notes/context
git fetch origin refs/notes/context:refs/notes/context
```

Add to `.git/config`:
```ini
[remote "origin"]
    fetch = +refs/notes/context:refs/notes/context
    push  = refs/notes/context:refs/notes/context
```

## Integration with Existing Git Commands

### `git log` with context

Git's log formatting lives in [log-tree.c](../log-tree.c). Notes are already displayable:

```bash
git log --notes=context
```

This already works with the existing `git notes` machinery. The `--notes=<ref>` flag tells `git log` to look up `refs/notes/<ref>` for each commit and append the note.

A `git ctx log` wrapper can pretty-print the JSON into readable prose:

```
commit abc123
Author: agent (claude-sonnet-4-6)
Date:   Mon Jun 15 2026

    Switch ref iteration to reftable backend

    [context: architecture | confidence: 0.85]
    Rationale: packed-refs degrades to O(n) scan on repos with >100k refs.
    reftable uses a block index giving O(log n) lookup. Trade-off accepted:
    reftable binary format is not hand-editable, but tooling covers all
    edit paths.

    Alternatives rejected:
    - sqlite: external dep, not acceptable for Git core
    - keep packed-refs + cache layer: complexity without solving root cause
```

### `git show` with context

```bash
git show --notes=context abc123
```

Shows the diff AND the decision context that produced it. For code review, this is the key surface: you see what changed and why in one view.

## The Context Graph

Commits form a DAG. Context objects, linked via the `related` field, form a parallel context graph.

```
Commit graph:          Context graph:
                       
  A ← B ← C            ctx-A ← ctx-B
               \                      \
                D ←── ctx-C ←──────── ctx-D
```

You can traverse: "what chain of decisions led to this commit?" by following `related` SHAs backwards through context blobs. Since blobs are content-addressed, each `related` pointer is immutable and verifiable — it can't silently drift.

A future `git ctx trace <commit>` command could render this graph:

```bash
git ctx trace HEAD
# → Decision: chose merge-ort over recursive (ctx: 9f3a...)
#   └→ Prior decision: abstracted merge strategy interface (ctx: 2b1c...)
#      └→ Prior decision: identified O(n²) rename detection bug (ctx: af44...)
```

## Agent Hook: Automatic Context Capture

A `post-commit` hook (in [templates/hooks/](../templates/)) that fires the agent's context capture automatically:

```sh
#!/bin/sh
# .git/hooks/post-commit
# Agent sets AGENT_CONTEXT_JSON before running `git commit`.
# Hook fires after SHA is final — note is a sidecar, never mutates the commit.
if [ -n "$AGENT_CONTEXT_JSON" ]; then
    git notes --ref=context add -m "$AGENT_CONTEXT_JSON" HEAD
    unset AGENT_CONTEXT_JSON
fi
```

The agent sets `AGENT_CONTEXT_JSON` as an env var before running `git commit`. The hook fires after the commit object is written, so the SHA is final. The note attaches to that final SHA. Commit never mutates.

For manual use, a `prepare-commit-msg` hook variant can open an editor with a context template:

```sh
#!/bin/sh
# .git/hooks/prepare-commit-msg  (extended for context)
# After writing commit message, write context to .git/COMMIT_CONTEXT
cat >> "$1" <<'EOF'

# --- Agent Context (will be stored as git note, not in commit message) ---
# decision_class: architecture | algorithm | interface | tradeoff | bug-fix
# rationale: 
# alternatives: 
EOF
```

A wrapper script strips the context section from the commit message, formats it as JSON, and attaches it as a note after the commit.

## Why Not Embed Context in the Commit Message?

Commit messages are part of the commit object. Changing the message changes the SHA. This is fine for small structured trailers (author, signed-off-by) but bad for rich freeform context because:

1. Rebasing replays commits — context would need to be carried through every rebase
2. Long context bloats `git log` for everyone, always
3. Context referencing other context (the `related` graph) requires stable SHAs, which commit messages can't provide without forward references

Notes are a sidecar. They travel separately. You opt in to seeing them. The commit object stays clean.

---

## What This Changes in the Codebase

This is designed to require **no changes to Git core** for v1. Everything runs on existing machinery:

| What | How | Existing code |
|------|-----|---------------|
| Store context blobs | `git hash-object` | [object-file.c](../object-file.c) |
| Attach to commit | `git notes --ref=context add` | [builtin/notes.c](../builtin/notes.c) |
| Push/fetch | standard refspec | [transport.c](../transport.c), [remote.c](../remote.c) |
| Show in log | `git log --notes=context` | [log-tree.c](../log-tree.c) |
| Pack for efficiency | automatic | [pack-objects.c](../pack-objects.c) |

New code needed: only `git ctx` as a shell script (lives in `contrib/context/`) and the hook templates.

## PR / Code Review Integration

When opening a PR, context notes for all commits in the branch can be gathered and surfaced as PR description scaffolding:

```bash
git ctx log origin/main..HEAD --format=pr-body
```

Output:
```markdown
## Decision log

**commit abc123** — Switch ref iteration to reftable backend
> Rationale: O(log n) vs O(n) read performance at scale. Alternatives
> considered: sqlite (external dep), packed-refs cache (hides root cause).

**commit def456** — Add block index for reftable reader
> Rationale: Required by the O(log n) guarantee. Block index lives in
> the same file format, no separate file to keep in sync.
```

The PR author didn't write this — the agent wrote it at decision time. The reviewer gets the thinking, not just the output.

## What v2 Adds

v1 is static — context is written at commit time and read later. v2 makes `git ctx` an **active participant in the agent loop**:

- **Checkpoints** (`git ctx checkpoint`) — intra-session tracking before a commit exists, capturing exploration and dead ends in real time
- **Agent-prompt output** (`git ctx log --format=agent-prompt`) — feeds prior decisions and open uncertainties back into the agent at session start
- **Uncertainty threading** — `uncertainty` fields from v1 schema surface across sessions as warnings before touching related files
- **`git ctx trace <file>`** — file-centric decision history, not just commit-centric (v1 has `trace <commit>`, v2 adds file path resolution)
- **`git ctx session-summary`** — end-of-session report: what was decided, avg confidence, open threads → feeds next session's starting context
- **Decision guard** — warning before the agent overwrites a high-confidence prior decision with a low-confidence new one

See [v2.md](v2.md) for the full design.

---

## Summary

The core idea in one sentence: **context is just another blob in the object database, attached to commits via `git notes`, forming a context graph that parallels the commit graph.**

No new object types. No SHA mutation. No workflow changes for teams that don't want it. Fully transportable with standard push/fetch. And context objects get the same integrity guarantees as source code — a context blob SHA is an immutable, verifiable reference to a specific decision at a specific moment in time.

The thing Git is good at — being the source of truth for change history — becomes the source of truth for decision history too.
