# v2: The Agent Loop — Context as Active Memory

v1 established the storage primitive: context is a JSON blob in Git's object database, attached to commits via `refs/notes/context`. Read [v1.md](v1.md) first.

v2 answers the next question: **what does the agent do with that context?**

The answer is that `git ctx` stops being a passive logger and becomes an active participant in the agent's decision loop. Context flows *into* the agent at the start of each session, not just *out* of it at commit time.

---

## The Problem v1 Doesn't Solve

v1 context is written at commit time. That means:

- The agent has already finished thinking by the time context is recorded
- The next session starts cold — the agent re-reads source files but not its own reasoning
- Dead ends explored mid-session are never recorded (only the conclusion is)
- An agent can confidently overturn a prior high-confidence decision without even knowing it did so

v2 closes those gaps.

---

## The Agent Loop: Before and After

**Without `git ctx` (v1 or nothing):**
```
session start → agent acts → commit → session end
                                 ↑
                        reasoning evaporates
```

**With v2:**
```
session start
    │
    ├── git ctx log --format=agent-prompt
    │       reads prior decisions + open uncertainties → fed to agent
    │
    agent works
    │   └── Claude Code writes thinking blocks to transcript JSONL  (automatic)
    │
    ├── git ctx checkpoint  (optional: explicit dead-end markers)
    ├── git ctx trace <file>  (before modifying a previously-decided file)
    │
    agent runs `git commit`
    │
    ├── PreToolUse hook fires → reads transcript → extracts all thinking
    │       → writes .git/CTX_THINKING
    ├── prepare-commit-msg hook → summarizes .git/CTX_THINKING → AGENT_CONTEXT_JSON
    ├── commit object written (SHA final)
    └── post-commit hook → stores raw thinking as blob → attaches note to SHA
    │
    session end
    │
    └── git ctx session-summary → open uncertainties + dead ends → next session seed
         ↑__________________________________________________________________|
                         context persists across sessions
```

The loop becomes self-aware. Each session builds on the reasoning of the last.

## New Concept 1: Checkpoints

v1 only captures context at commit time. But the most valuable reasoning happens *during* a session — when the agent is still deciding, still exploring, possibly hitting dead ends.

**`git ctx checkpoint`** records a context snapshot *before* a commit exists. Checkpoints live in `.git/CTX_SESSION/` (a temp directory scoped to the current session), not in the object DB yet.

```bash
# Agent is evaluating options, not yet committed to one
git ctx checkpoint --mode=deciding \
  --note "evaluating reftable vs packed-refs for ref iteration"

# Agent spikes an alternative and abandons it
git ctx checkpoint --mode=exploring \
  --note "tried sqlite as ref store — hit external dep constraint, abandoning"

# Agent commits with the full journey assembled automatically
git ctx add --exploration-path "$(git ctx checkpoints --this-session)"
```

At commit time, `git ctx add` reads all checkpoints from the session and embeds them as `exploration_path` in the context blob. The agent doesn't reconstruct the journey at the end — it was recorded as it happened.

### Updated schema with exploration_path

```json
{
  "v": 2,
  "kind": "decision",
  "mode": "implementing",
  "summary": "Switch to reftable backend",
  "rationale": "O(log n) vs O(n). Block index makes random access constant.",
  "decision_class": "architecture",
  "confidence": 0.85,
  "uncertainty": "concurrent write behavior under multi-writer scenario untested",
  "affects": ["refs/reftable-backend.c"],
  "related": ["<sha-of-prior-context-blob>"],
  "exploration_path": [
    { "mode": "deciding",   "note": "evaluating reftable vs packed-refs" },
    { "mode": "exploring",  "note": "tried sqlite — external dep, abandoned" },
    { "mode": "deciding",   "note": "confirmed reftable satisfies all constraints" }
  ]
}
```

The `exploration_path` is a compressed journal of how the decision was reached. Future agents reading this don't re-explore dead ends — they see that sqlite was already tried and why it was rejected.

## New Concept 2: Passive Thinking Capture via Claude Code's Transcript

Checkpoints require the agent to explicitly call `git ctx checkpoint`. There is a fully automatic path that requires zero agent action: Claude Code already stores every thinking block in a session transcript JSONL file, and every hook receives the path to that file. We can intercept it at the exact moment the agent runs `git commit`.

### The proof: thinking blocks are in the transcript

Claude Code writes the full session to a JSONL file at:

```
~/.claude/projects/<project-path>/<session-id>.jsonl
```

Each line is a JSON object. Assistant turns look like this:

```jsonl
{
  "type": "assistant",
  "message": {
    "content": [
      {
        "type": "thinking",
        "thinking": "The user wants me to switch the ref backend. Let me evaluate reftable vs packed-refs. packed-refs degrades to O(n) scan on repos with large ref counts. reftable uses a block index giving O(log n)...",
        "signature": "EqkDCmUID..."
      },
      {
        "type": "text",
        "text": "I'll switch to the reftable backend because..."
      }
    ]
  }
}
```

Every thinking block the agent produced across the entire session is there, in order, verbatim. The `signature` field is Claude Code's integrity seal on the thinking content.

Every hook receives `transcript_path` in its input JSON pointing directly to this file.

### The capture hook: `PreToolUse` on `git commit`

Claude Code fires `PreToolUse` before every tool call. When the tool is `Bash` and the command is `git commit`, that is the exact right moment to extract all thinking from the session transcript and write it to `.git/CTX_THINKING` before the commit object exists.

Configure the hook in `.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash /path/to/ctx-capture.sh"
          }
        ]
      }
    ]
  }
}
```

The hook script:

```bash
#!/bin/bash
# ctx-capture.sh — runs before every Bash tool call

INPUT=$(cat)
CMD=$(echo "$INPUT"    | jq -r '.tool_input.command')
TRANSCRIPT=$(echo "$INPUT" | jq -r '.transcript_path')

# only act when the agent is about to commit
if ! echo "$CMD" | grep -qE "^git commit"; then
    exit 0
fi

# extract all thinking blocks from this session, in order
jq -r '
  select(.type == "assistant") |
  .message.content[] |
  select(.type == "thinking") |
  .thinking
' "$TRANSCRIPT" > .git/CTX_THINKING

exit 0   # allow the commit to proceed
```

At this point `.git/CTX_THINKING` contains the agent's full internal reasoning for the session — every decision point, every dead end considered, every uncertainty weighed — extracted automatically with no agent involvement.

### Summarization via `prepare-commit-msg`

Git fires `prepare-commit-msg` between "commit invoked" and "commit object written." The hook reads `.git/CTX_THINKING` and calls a summarizer — the same or a lighter model — with a narrow extraction prompt:

```sh
#!/bin/sh
# .git/hooks/prepare-commit-msg
if [ -f .git/CTX_THINKING ]; then
    AGENT_CONTEXT_JSON=$(git-ctx-summarize < .git/CTX_THINKING)
    export AGENT_CONTEXT_JSON
fi
```

`git-ctx-summarize` does not generate new reasoning. It extracts `summary`, `rationale`, `decision_class`, `mode`, `alternatives`, `uncertainty`, and `confidence` from what the agent already produced. The raw thinking is the source of truth; the summary is the readable index into it.

### The raw thinking is stored as a Git blob

Before clearing `.git/CTX_THINKING`, `post-commit` writes it into the object database:

```sh
#!/bin/sh
# .git/hooks/post-commit
if [ -n "$AGENT_CONTEXT_JSON" ]; then
    RAW_SHA=$(git hash-object -w .git/CTX_THINKING)
    AGENT_CONTEXT_JSON=$(echo "$AGENT_CONTEXT_JSON" \
        | jq --arg r "$RAW_SHA" '. + {raw_thinking: $r}')
    git notes --ref=context add -m "$AGENT_CONTEXT_JSON" HEAD
    rm -f .git/CTX_THINKING
    unset AGENT_CONTEXT_JSON
fi
```

The context note carries a `raw_thinking` field pointing to the full blob. The summary is what you read every day. The blob is there to audit exact reasoning, diff how thinking changed across sessions, or re-summarize with a better model in the future.

### Updated schema with raw_thinking

```json
{
  "v": 2,
  "kind": "decision",
  "mode": "implementing",
  "summary": "switch to reftable backend",
  "rationale": "packed-refs degrades to O(n) at scale. reftable block index gives O(log n). trade-off: binary format not hand-editable, tooling covers all edit paths.",
  "decision_class": "architecture",
  "confidence": 0.85,
  "uncertainty": "concurrent writes under multi-writer scenario untested",
  "affects": ["refs/reftable-backend.c"],
  "related": ["<sha-of-prior-context-blob>"],
  "raw_thinking": "<sha-of-CTX_THINKING-blob>",
  "exploration_path": [...]
}
```

### The full passive flow

```
agent works across the session
    → Claude Code writes thinking blocks to transcript JSONL (automatic)

agent runs `git commit` via Bash tool
    ↓
Claude Code fires PreToolUse hook (before commit executes)
    → ctx-capture.sh reads transcript_path
    → extracts all thinking blocks with jq
    → writes to .git/CTX_THINKING
    ↓
git commit executes
    ↓
Git fires prepare-commit-msg hook
    → git-ctx-summarize reads .git/CTX_THINKING
    → extracts structured fields → sets AGENT_CONTEXT_JSON
    ↓
commit object written — SHA is final
    ↓
Git fires post-commit hook
    → writes .git/CTX_THINKING as blob → stores SHA
    → attaches note to final commit SHA
    → deletes .git/CTX_THINKING
```

The agent does nothing. It just works and commits. Three hooks do the rest.

### What is and isn't in the transcript

The transcript contains:
- **`type: "thinking"`** — the agent's full internal reasoning, verbatim, for every turn
- **`type: "text"`** — the agent's visible text responses
- **`type: "tool_use"`** — every tool call the agent made
- **`type: "tool_result"`** — every tool result

What is not in the transcript: the thinking tokens Claude Code chose not to show (there is a display budget, but the transcript stores the full untruncated thinking content regardless of what the UI renders).

### Relationship to checkpoints

Passive capture and checkpoints are complementary:

- **Passive capture** extracts the full thinking stream automatically — no agent action, no missed reasoning
- **Checkpoints** are for explicit structural markers the agent wants to emphasize at specific moments ("I abandoned sqlite here") that would require the summarizer to parse out from prose

In a well-instrumented session: passive capture handles everything, checkpoints add structure at key inflection points the agent wants to surface cleanly.

## New Concept 3: Agent-Prompt Output Format

At the start of a session, the agent needs to consume prior context in a format it can act on — not a pretty log for humans.

```bash
git ctx log --last=10 --format=agent-prompt
```

Output (structured text the agent reads directly):

```
PRIOR DECISIONS (most recent first):

[abc123 | implementing | confidence:0.85]
Summary: Switched to reftable backend
Rationale: O(log n) vs O(n) at scale. Block index gives random access.
Uncertainty: concurrent writes under multi-writer scenario unresolved
Affects: refs/reftable-backend.c

[9fa312 | debugging | confidence:0.60 | reversals:3]
Summary: Fixed concurrent write race
Rationale: Added lock around ref iteration
Uncertainty: fix holds under single-writer assumption only — not stress tested
Affects: refs/reftable-backend.c

OPEN UNCERTAINTIES (unresolved across all sessions):
→ [refs/reftable-backend.c] concurrent ref writes under multi-writer scenario
→ [refs/reftable-backend.c] performance assumption untested beyond 500k refs

EXPLORATION DEAD ENDS (don't re-try these):
→ sqlite as ref store: ruled out — external dependency constraint
→ packed-refs cache layer: ruled out — hides root cause, doesn't fix O(n)
```

The agent now knows:
1. What decisions were already made and why
2. What was left unresolved (and where)
3. What alternatives were already ruled out — so it doesn't re-derive them

This is institutional memory, encoded in the repo, available to any agent or human who clones it.

## New Concept 4: Uncertainty Threading

The `uncertainty` field from v1's schema becomes load-bearing in v2.

When a context note has a non-empty `uncertainty`, it's tracked in a separate index:

```
refs/notes/context-uncertainty
```

This ref maps files → list of unresolved uncertainty SHAs. Before the agent touches a file, `git ctx` checks this index:

```bash
git ctx trace refs/reftable-backend.c
```

Output:

```
DECISION HISTORY FOR: refs/reftable-backend.c

[abc123 | implementing | confidence:0.85 | deep-thinking]
"Switch to reftable backend"
Rationale: O(log n) lookup at scale
⚠ OPEN UNCERTAINTY: concurrent writes under multi-writer scenario

[9fa312 | debugging | confidence:0.60 | 3 reversals]
"Fix concurrent write race"
⚠ OPEN UNCERTAINTY: fix holds under single-writer assumption only

RECOMMENDATION: This file carries 2 unresolved uncertainties from prior sessions.
Match or exceed prior thinking depth before modifying.
```

Uncertainties are never silently dropped. They surface as file-level annotations every time the file is about to be touched. The agent either resolves them, defers them explicitly, or escalates thinking depth.

### Resolving an uncertainty

When an uncertainty is finally addressed, the agent records the resolution:

```bash
git ctx resolve abc123 \
  --resolves "concurrent writes under multi-writer scenario" \
  --resolution "stress tested under 16-writer benchmark, no races found" \
  --confidence 0.95
```

This writes a new context blob with `kind: "resolution"` and `resolves: "<sha-of-original-uncertainty>"`, removes the file from the uncertainty index, and links the resolution into the context graph via `related`.

## New Concept 5: Decision Guard

Before the agent overwrites a prior decision, v2 surfaces a structured warning:

```
⚠ DECISION GUARD
  You are about to modify refs/reftable-backend.c.
  
  Prior decision (abc123 | confidence: 0.85 | deep-thinking | 0 reversals):
  "Switch to reftable backend"
  Rationale: O(log n) vs O(n) at scale
  
  Your current confidence: 0.60
  Prior confidence: 0.85
  
  Confidence is lower than prior decision.
  Recommend: deepen analysis or explicitly defer with justification.
  
  Options:
    (1) proceed — attach context explaining the reversal
    (2) defer — record intent to revisit, don't touch file yet
    (3) escalate — go back to deciding mode before modifying
```

The guard is triggered by `git ctx trace` returning a higher-confidence prior decision than the agent's current mode. It doesn't block — it creates friction and demands explicit justification.

If the agent proceeds, the new context note links to the prior one via `related` and includes a `reversal_of` field:

```json
{
  "kind": "decision",
  "reversal_of": "<sha-of-prior-context-blob>",
  "summary": "Revert to packed-refs for single-writer repos",
  "rationale": "reftable overhead not justified below 10k refs",
  "confidence": 0.75
}
```

This makes reversals visible in `git ctx log` as a decision thread — not a rewrite of history.

## New Concept 6: Session Summary

At session end, `git ctx session-summary` produces a report that serves two purposes: retrospective for the human, starting context for the next agent session.

```bash
git ctx session-summary
```

```
SESSION SUMMARY — 2026-06-15

Commits this session: 4
Decisions recorded: 3 architecture, 1 debugging
Avg confidence: 0.78
Reversals: 1 (9fa312 → overturned abc123 partial fix)

RESOLVED THIS SESSION:
✓ packed-refs O(n) performance — resolved by switching to reftable

OPEN UNCERTAINTIES (carried forward):
→ [refs/reftable-backend.c] concurrent writes under multi-writer scenario
→ [refs/reftable-backend.c] performance cliff beyond 500k refs

EXPLORATION DEAD ENDS (don't re-explore):
→ sqlite as ref store (external dep constraint)
→ packed-refs cache layer (hides root cause)

SUGGESTED STARTING CONTEXT FOR NEXT SESSION:
  git ctx log --last=5 --format=agent-prompt
  Priority file: refs/reftable-backend.c (2 open uncertainties)
```

The "Suggested Starting Context" block is machine-readable. The next session's init script runs it verbatim. The uncertainty thread never silently drops between sessions.

## The Reversals Counter

The `reversals` field in `--format=agent-prompt` output (`[9fa312 | debugging | confidence:0.60 | reversals:3]`) is computed, not stored. `git ctx` counts how many times a context blob has been overturned by following `reversal_of` chains forward through the context graph.

High reversal count = this decision space is unstable. Treat it as a signal:
- High reversals + low confidence = don't touch without deep thinking mode
- High reversals + high confidence = stable now, but watch for regression

## Storage Additions for v2

v1 storage: `refs/notes/context` (one note per commit).

v2 adds two more lightweight structures:

| Location | Contents | Lifetime | Purpose |
|----------|----------|----------|---------|
| `refs/notes/context` | commit SHA → context blob | permanent | v1, unchanged |
| `refs/notes/context-uncertainty` | file path → uncertainty blob SHAs | permanent | fast lookup for decision guard |
| `.git/CTX_SESSION/` | checkpoint files | ephemeral (cleared at commit) | intra-session explicit markers |
| `.git/CTX_THINKING` | thinking blocks extracted from transcript | ephemeral (cleared at commit) | raw input to summarizer |
| `~/.claude/projects/…/<session>.jsonl` | Claude Code's live session transcript | managed by Claude Code | source of truth for thinking blocks |
| object DB (loose/pack) | raw thinking blobs | permanent | full reasoning audit trail |

`.git/CTX_THINKING` is populated by the `PreToolUse` hook reading the Claude Code session transcript at the moment `git commit` runs. It follows the same gitdir temp file pattern as `COMMIT_EDITMSG` and `MERGE_MSG` — exists between operations, cleaned up after.

The session directory (`.git/CTX_SESSION/`) is ephemeral — it doesn't live in the object DB until `git ctx add` at commit time reads checkpoints, assembles `exploration_path`, and writes the final blob. After the blob is written, `.git/CTX_SESSION/` is cleared.

## What the Full v2 `git ctx` Surface Looks Like

```bash
# Session start
git ctx log --last=10 --format=agent-prompt   # feed prior context to agent

# During session
git ctx checkpoint --mode=deciding --note "..."    # record mid-session thinking
git ctx checkpoint --mode=exploring --note "..."   # record a dead end
git ctx trace refs/some/file.c                     # check prior decisions for file

# At commit
git ctx add                                        # assemble from checkpoints + attach
# (or via post-commit hook automatically)

# Uncertainty management
git ctx resolve <context-sha> --resolution "..."   # close an open uncertainty

# Session end
git ctx session-summary                            # retrospective + next-session seed

# Sync
git ctx push origin
git ctx fetch origin
```

## What Still Doesn't Change in Git Core

v2 is still zero changes to Git core. The additions:

| What | How | Where the code lives |
|------|-----|----------------------|
| Uncertainty index | `git notes --ref=context-uncertainty` | [builtin/notes.c](../builtin/notes.c) |
| Session checkpoints | files in `.git/CTX_SESSION/` | gitdir filesystem (like MERGE_MSG) |
| Thinking extraction | `PreToolUse` hook reads `transcript_path` | `.claude/settings.json` hook config |
| Thinking summarization | `prepare-commit-msg` hook + `git-ctx-summarize` | [templates/hooks/](../templates/) |
| Raw thinking blob | `git hash-object -w .git/CTX_THINKING` | [object-file.c](../object-file.c) |
| Note attachment | `post-commit` hook | [templates/hooks/](../templates/) |
| Resolution notes | context blobs with `kind: "resolution"` | same object DB |
| Reversal chain | `reversal_of` field + graph traversal | blob content, no new refs |

The only new code is in `contrib/context/git-ctx` — a shell script that knows how to assemble all of this from the existing plumbing.

| Capability | v1 | v2 |
|---|---|---|
| Context stored at commit time | ✓ | ✓ |
| Context pushed/fetched with repo | ✓ | ✓ |
| PR decision log auto-generated | ✓ | ✓ |
| Agent reads prior context at session start | — | ✓ |
| Passive thinking capture via Claude Code transcript + PreToolUse hook | — | ✓ |
| Raw thinking stored as blob, reachable from note | — | ✓ |
| Mid-session checkpoints (dead ends, exploration) | — | ✓ |
| Uncertainty threaded across sessions | — | ✓ |
| Decision guard before overturning high-confidence decisions | — | ✓ |
| File-centric decision history | — | ✓ |
| Session summary → next session seed | — | ✓ |
| Reversal chain (decision thread, not history rewrite) | — | ✓ |

v1 makes context *durable*. v2 makes it *active* — a feedback loop that closes the gap between what the agent decided and what the next agent knows when it arrives.
